# MinerU SaaS 与开源能力映射

## 为什么要单独做这份对照

团队在对齐知识库时，最容易踩的坑之一就是把“开源引擎支持”直接等同于“SaaS 页面现在怎么表现”。

这其实是两层东西：

- 开源层：底层引擎、部署、模型、接口、输出
- SaaS 层：在线 API、网页端、客户端、上传流程、页面展示、结果交互

尤其在 `3.0 / 3.1` 之后，Office 原生解析、图表解析、模型升级都需要产品层同步适配。

## 一句话结论

- `MinerU` 开源版更像能力底座
- `MinerU SaaS` 更像面向业务用户的产品化封装
- 两者有关联，但不应被写成同一层

## 快速对照

| 维度 | 开源主线 | SaaS / 在线产品层 | 当前知识库建议 |
|---|---|---|---|
| 目标 | 私有化、部署、二开、服务化 | 零部署、在线使用、网页/API/客户端体验 | 先分清“引擎”与“产品” |
| 核心入口 | `opendatalab/MinerU` | `mineru.net`、在线 API、客户端 | 对外说法不要混写 |
| 接入层 | 本地 CLI、FastAPI、Router、Gradio、OpenAI-compatible | 在线 API、CLI/SDK、MCP、网页端、客户端 | 研发看接入路径，运营看产品入口 |
| 版本演进 | 以仓库 README / changelog 为准 | 以产品页面、API docs、需求适配为准 | 两边都要核 |
| 页面表现 | 由部署和输出决定 | 受前端交互和产品策略影响 | SaaS 不等于原始输出全量直出 |

## 文件格式支持

### 开源主线当前可稳定写法

根据 `2026-04-22` 核对的官方 `MinerU` 仓库 README：

- 支持 `PDF`
- 支持图片
- 支持 `DOCX`
- 支持 `PPTX`
- 支持 `XLSX`

### SaaS 产品层当前更谨慎的写法

根据内部文档 [`MinerU SaaS 产品优化 —— 适配 3.0_3.1 开源版本改造`](</Users/wangshasha/Desktop/MinerU SaaS 产品优化 —— 适配 3.0_3.1 开源版本改造.docx>)：

- `DOCX / PPTX / XLSX` 需要同步产品层适配
- `DOC / PPT / XLS` 可能通过后台转换后再进入新版 Office 解析流程
- API 文档、上传组件、格式提示、结果页都需要同步更新

因此建议知识库中的写法是：

- `开源主线已支持主流 Office 原生解析`
- `SaaS 侧是否完全体现在页面与文档上，应以当前产品页和 API docs 为准`

## 模型与模式选择

| 场景 | 开源主线理解 | SaaS 产品层理解 |
|---|---|---|
| PDF / 图片 | 可以区分 `pipeline`、`hybrid`、`vlm` 等后端或模型路线 | 页面上通常会暴露模型 / 模式选择 |
| Office 文件 | 更偏原生解析路线，不一定需要用户干预 | 内部文档建议 Office 上传后隐藏或置灰模型选择 |

知识库建议写法：

- `Office 文档更适合被理解为原生解析路径，而不是让用户显式切换 PDF 模型。`

## 输出与可视化差异

### 开源主线

开源部署通常可以获得：

- Markdown
- JSON
- 中间结果
- 布局可视化
- 更细的本地输出文件

### SaaS 产品层

根据内部产品适配文档，Office 场景下需要特别注意：

- 结果不包含 `bbox`
- 不支持布局框可视化
- 不支持原文与解析结果联动
- 不支持文本编辑
- 更适合只展示解析后的 Markdown / 结构化内容

这意味着：

- 不能把“开源里有布局可视化”直接写成“所有 SaaS 结果页都支持布局联动”

## 图表能力

### 开源 / 模型层

官方仓库 `3.1.0` 更新说明已明确：

- 主模型升级后支持图像与图表解析
- 支持复杂版面增强能力

### SaaS 产品层

内部产品文档建议：

- 图表解析采用 `Beta` 弱展示策略
- 不单独做功能主打
- 结果展示里需要提示用户自行校验

知识库建议写法：

- `图表解析已进入能力范围，但在产品层更适合以 Beta 能力呈现。`

## API 与产品页面的区别

| 维度 | 开源服务化 | 在线 API / SaaS |
|---|---|---|
| 调用方式 | 本地服务、Docker、Compose、OpenAI-compatible | 官方 REST API、CLI、SDK、MCP |
| 限制来源 | 本地算力、后端配置、部署方式 | 官方 live docs、账号额度、产品策略 |
| 文档入口 | GitHub README / docs | `mineru.net/apiManage/docs` |

知识库建议：

- 只要涉及文件大小、页数、额度、限频，优先看 online API docs
- 只要涉及部署、后端、服务编排，优先看开源仓库 README / docs

## 当前最值得提醒团队的几个点

### 1. Office 原生解析不等于 Office 全量产品体验一致

开源支持 Office 原生解析，不代表网页端已经在所有位置同步完成交互和结果展示优化。

### 2. 图表能力不等于应该强宣传

底层支持图表解析，不代表产品层应该把它当成熟核心卖点去强推。

### 3. 许可证和额度都要分来源

- 许可证优先看开源主仓库
- API 上限与额度优先看 live docs

### 4. 开源适配和 SaaS 发布节奏并不总是同一天完成

研发、产品、运营对齐知识库时，需要接受一个现实：

- 开源先升级
- SaaS 再适配
- 文档与前端文案可能最后补齐

## 推荐对外说法

### 推荐

- `MinerU 同时提供开源部署能力和在线产品能力。`
- `开源主线负责底层解析与部署能力，SaaS 提供面向业务用户的在线 API、网页端和客户端体验。`
- `对于 Office、图表等新能力，建议区分“底层已支持”和“当前产品层展示方式”两种口径。`

### 不推荐

- `只要开源支持，网页端一定已经完全一样`
- `API 文档、网页端、开源仓库对所有细节一定天然同步`
