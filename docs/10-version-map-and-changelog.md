# MinerU 版本地图与变更记录

## 为什么要单独做版本文档

MinerU 现在已经不是一个“只有 PDF 解析”的单点工具，产品层、模型层、SaaS 层、生态层都在持续变化。

如果不把版本拆开，团队很容易出现这些问题：

- 产品稿把旧口径当新能力
- 研发按开源版本理解 SaaS 页面表现
- 运营把课件中的额度、许可、支持格式直接写成当前事实
- 开源协作者分不清 `3.0` 和 `3.1` 的能力边界

这份文档专门记录版本演进。

## 当前可确认的关键版本节点

| 版本 | 日期 | 主要来源 | 这版最重要的变化 |
|---|---|---|---|
| `3.1.0` | `2026-04-18` | 官方 `MinerU` 仓库 README | 许可证升级、VLM 主模型升级到 `MinerU2.5-Pro-2604-1.2B`、新增 `PPTX/XLSX` 原生解析 |
| `3.0.0` | `2026-03-29` | 官方 `MinerU` 仓库 README | `DOCX` 原生解析、`pipeline` 后端升级、`API/CLI/Router` 编排升级 |
| `v2.x` | 历史阶段 | 官方 README + 训练营资料 + 既有认知 | 以 PDF / 图片解析为主，生态和模型形态仍在快速演化 |

说明：

- `3.1.0` 和 `3.0.0` 的日期、能力点来自 `2026-04-22` 核对到的官方 `MinerU` 仓库 README
- `v2.x` 这里只做阶段说明，不在本文件里展开全部子版本差异

## `3.1.0` 可以稳定写进知识库的点

### 官方仓库已明确写出的变化

- `2026-04-18` 发布 `3.1.0`
- 项目许可证从 `AGPLv3` 切换为基于 `Apache 2.0` 的 `MinerU Open Source License`
- VLM 主模型升级为 `MinerU2.5-Pro-2604-1.2B`
- 新模型支持图像与图表解析、截断段落合并、跨页表格合并、表格内图片识别
- 新增 `PPTX` 与 `XLSX` 原生解析
- 由此补齐图片、`PDF`、`DOCX`、`PPTX`、`XLSX` 的全格式主流覆盖

### 对团队最有实际意义的影响

- 产品同学可以把“全格式原生解析”作为 `3.1` 之后的重要能力点
- 运营同学在写格式覆盖时，建议加上“以 `3.1.0` 后官方仓库口径为准”
- 研发同学需要意识到 `PPTX/XLSX` 支持和更强的图表/图像能力，是当前主线版本的重要升级
- 开源协作者需要注意许可证已经发生变化，不能默认沿用旧 `AGPL` 口径

## `3.0.0` 可以稳定写进知识库的点

### 官方仓库已明确写出的变化

- `2026-03-29` 发布 `3.0.0`
- 正式支持 `DOCX` 原生解析
- 相比“先转 PDF 再解析”，官方强调端到端速度提升数十倍
- `pipeline` 后端升级，在 OmniDocBench (v1.5) 上达到 `86.2`
- `pipeline` 增加了表格内图片/公式、印章文字、竖排文本等复杂场景能力
- `mineru`、`mineru-api`、`mineru-router` 等工程编排能力增强

### 对团队最有实际意义的影响

- `3.0` 是“从 PDF 主线走向 Office 原生解析”的起点
- `3.0` 之后，知识库里就不能再把 MinerU 简化成“只做 PDF”
- 如果要解释 SaaS 为什么开始适配 Office 文件，这是 `3.0/3.1` 升级链条的一部分

## `v2.x -> 3.0 -> 3.1` 的简化理解

| 阶段 | 更像什么 | 对外表达重点 |
|---|---|---|
| `v2.x` | 以 PDF / 图片为主的高质量文档解析引擎 | 高精度解析、RAG/LLM 预处理、复杂版面能力 |
| `3.0` | 从 PDF 主线扩展到 Office 原生解析起点 | `DOCX` 原生解析、工程可用性升级 |
| `3.1` | 进入主流文档全格式原生覆盖阶段 | `PPTX/XLSX` 原生解析、VLM 升级、许可证升级 |

## SaaS 适配文档带来的补充信息

内部文档 [`MinerU SaaS 产品优化 —— 适配 3.0_3.1 开源版本改造`](</Users/wangshasha/Desktop/MinerU SaaS 产品优化 —— 适配 3.0_3.1 开源版本改造.docx>) 说明了一个很重要的事实：

- 开源版本升级后，SaaS 产品层并不是“天然自动等于开源能力”
- 网页端、上传入口、API 文档、结果页展示、提示文案都需要同步适配

从该文档可归纳出当前最值得沉淀的产品层变化：

- Office 文件上传后，前端不一定保留模型选择
- Office 原生解析结果不包含 `bbox`
- Office 场景下不支持布局框联动和编辑
- 图表解析应作为 `Beta` 能力弱展示
- `wmf/emf` 在部分 Linux 环境下存在渲染限制

这些内容不应直接写成“开源底层限制”，更适合写进 SaaS 能力映射和已知限制文档。

## 当前最需要防止的版本误写

### 1. 许可证

- 旧说法：`AGPL-3.0`
- 当前更可信来源：官方 `MinerU` 仓库 README 与 `LICENSE.md`
- 当前处理建议：涉及当前项目代码许可证时，优先写 `MinerU Open Source License`

### 2. 页数与额度

- live docs 当前显示：精准解析 `<= 200 页`
- 官方 `llms.txt` 仍写：精准解析 `<= 600 页`
- 当前处理建议：对外接入和 FAQ 以 live docs 为准，在知识库保留差异记录

### 3. 支持格式

- `3.1` 的官方仓库口径已明确 `PDF / 图片 / DOCX / PPTX / XLSX`
- SaaS 页面和 API 文案是否完全同步，需要继续按 live docs 和产品页面核验

## 推荐对外表达

### 推荐

- `MinerU 在 2026 年 3 月和 4 月连续完成 3.0 与 3.1 版本升级，从 DOCX 原生解析进一步扩展到 PPTX/XLSX 原生解析。`
- `当前官方仓库已将 3.1.0 作为主线版本，并记录了许可证升级与主模型升级。`

### 不推荐

- `MinerU 一直都是 AGPL-3.0`
- `MinerU 早就全量支持所有 Office 行为且各端表现完全一致`
- `SaaS 页面表现等于底层开源引擎能力`
