# MinerU KIE 使用指引整理版

## KIE 是什么

这里的 `KIE` 可以理解为 MinerU 的文档智能抽取能力，适合从发票、票据、表单等非结构化文档中提取关键字段，并输出结构化数据。

相比“只做解析”，KIE 更强调：

- 可配置流程
- 结构化字段抽取
- 面向业务结果，而不是只输出中间文本

## 典型用户

- 财务报销和票据整理
- 表单录入
- 合同 / 证照字段提取
- 运营资料结构化

## 整体流程

KIE 当前的流程是可组合的，核心有三个节点：

1. `解析`
2. `分割`
3. `抽取`

对新手最常见的两条路径：

- `解析 -> 抽取`
- `解析 -> 分割 -> 抽取`

## 1. 解析

解析是基础层，把 PDF / 图片中的内容先转成机器可读结构。

### 能做什么

- 识别文本、表格、图片
- 做版面分析
- 输出 Markdown 和 JSON

### 适合什么时候直接用默认配置

- 文档结构不复杂
- 主要想先看整体内容
- 还不需要做太细的字段抽取

## 2. 分割

分割层更偏“语义分组”，适合长文档或内容混杂的文档。

### 适合的场景

- 合同里既有正文又有附件
- 多页票据 / 混合资料
- 长报告中要拆不同章节或信息块

### 关键点

- 可以用自然语言描述分割策略
- 可以新增多个分组
- 分组描述越具体，后续抽取越稳

## 3. 抽取

抽取是在解析或分割的基础上，对目标字段做定向结构化。

### 核心能力

- 基于 Schema 定义数据结构
- 支持模型智能推荐 Schema
- 支持 `String` / `Number` / `Boolean` / `Date` / `Enum` / `Object` / `Array` / `Null`

### 更适合怎样理解

不是“再做一次 OCR”，而是：

- 先看懂文档
- 再按你要的字段结构把结果整理出来

## 文件限制与上传方式

根据你给的 KIE 指引文档，当前界面层的限制是：

- 支持格式：`PDF`、`JPG`、`PNG`
- 单文件最大：`10MB`
- PDF 最多：`10 页`
- 批量最多：`10 个文件`

这套限制更像当前产品体验层约束，不等同于 MinerU Open API 的整体上限。

## 部署为 API

KIE 流程可以部署成 API 服务，这一点很关键，因为它意味着 KIE 不只是页面工具，还可以进入自动化系统。

### 部署前条件

- 流程里包含解析模块
- 至少上传并运行过一次
- 所有已配置模块都至少成功跑过一次

### 部署后会得到

- `Pipeline ID`
- API 端点
- 调用示例

## 怎么把 KIE 写进知识库口径

推荐写法：

- `MinerU 不只是文档解析，也在向文档智能抽取延伸`
- `KIE 更适合发票、票据、表单、合同字段等结构化抽取场景`

不推荐写法：

- `KIE 已经等于完整通用工作流平台`
- `KIE 没有任何文件限制`
